## 안전 원칙 (전 Stage 공통, 필독)

- 순서 역전 금지: **익일 재확인 → 페이퍼(G3) → 소액 1주(G4) → 소액 20거래일(G5) → 본격 운용**
- 각 Stage는 판정 조건 **전 항목 충족 후** 다음 Stage 착수. 부분 충족 이월 금지
- 게이트 판정은 체크리스트 문서화 후 **익일 재확인** (당일 판정 금지 ― 확증편향 방지)
- 오류·사고 발생 시: 원인 규명 전 재개 금지, 기간 카운트는 **처음부터 재산정**
- 운용 중 동일 계좌 **수동 매매 금지** (정합성 불일치 → 자동 거래 중단의 최다 원인)
- 클라이언트당 유효 토큰 1개: **봇 가동 중 check_* 스크립트 실행 금지** (봇 토큰 무효화됨)
- 본 계획은 수익을 보장하지 않으며, 실거래 손실 책임은 운용자 본인에게 귀속

## 전체 로드맵

```
Stage 0 (D+1, 1일)      게이트 익일 재확인 ─ G1·G2 확정
Stage 1 (D+1~2, 1일)    운용 준비 ─ Telegram 알림·paper 전환·가동 방식 확정
Stage 2 (2주+)          G3: 페이퍼트레이딩 연속 10거래일 무오류
Stage 3 (1~3일)         G4: 소액 실거래 1주 완전 사이클 + 거래세율 확정
Stage 4 (4주+)          G5: 소액 운용 20거래일 무사고 → 본격 운용 승인
Stage 5 (병행)          상시 운영 체계 전환 (systemd 서버 ― 선택)
```

주의: Stage 2·4의 기간은 **거래일 기준**이며 오류 발생 시 재산정되므로, 달력상 최소 6~8주를 상정할 것. 서두르지 않는 것이 이 단계의 핵심 역량임.

## Stage 1. 운용 준비 (Telegram·paper 전환·가동 방식)

### 1.1 목표

페이퍼 운용을 "지켜볼 수 있는" 상태로 만든다. 알림 없이 가동하면 10거래일 판정 증적(일일 리포트 확인)이 부실해진다.

### 1.2 단계별 절차

| Step | 작업 | 상세 절차 | 확인 사항 |
|---|---|---|---|
| 1-1 | Telegram 봇 생성 | ① 텔레그램에서 `@BotFather` 검색 → `/newbot` → 봇 이름·아이디 입력 ② 발급된 **봇 토큰** 복사 (1회 노출 ― 시크릿과 동일하게 취급) | 토큰을 메신저·메모에 평문 저장 금지 |
| 1-2 | chat_id 확인 | ① 생성한 봇에게 아무 메시지 1회 전송 ② 브라우저에서 `https://api.telegram.org/bot<토큰>/getUpdates` 접속 ③ 응답 JSON의 `message.chat.id` 값 확인 | 숫자 ID (예: 123456789) |
| 1-3 | .env 등록 | `.env`에 `TELEGRAM_BOT_TOKEN=` / `TELEGRAM_CHAT_ID=` 기재 | `git status`에 .env 미표시 재확인 |
| 1-4 | 알림 테스트 | `python -m src.orchestrator --once` 실행 → 폰으로 "▶ 기동 (mode=dry_run)" / "■ 종료" 수신 확인 | 미수신 시 토큰·chat_id 재확인 (NullNotifier 로그가 뜨면 .env 미로딩) |
| 1-5 | paper 전환 | `config/settings.yaml`의 `runtime.mode`를 `paper`로 변경 ― **이 1곳만** | 다른 파일·코드 수정 금지 |
| 1-6 | 한도 확인 | `limits` 섹션 점검: `daily_loss_limit`(페이퍼 기준으로도 유지), `max_order_amount`, `max_position_pct` | 페이퍼여도 실전과 동일 한도로 운용 (검증 목적) |
| 1-7 | 가동 방식 확정 | 아래 A/B 중 택1 | ― |

**가동 방식 선택**

| 방식 | 내용 | 권장 대상 |
|---|---|---|
| A. 로컬 PC 수동 가동 | 장 시작 전 `python -m src.orchestrator` 실행, 장 종료 후 Ctrl+C(안전 종료) | G3 단계 권장 ― 관찰이 쉬움. 단, 10거래일 동안 매일 가동해야 함 |
| B. 상시 서버 (systemd) | `docs/operations.md` 1장 절차로 VM/미니PC 배포 | G4 이후 권장. **서버 공인 IP를 포털에 추가 등록** 필수 |

주의(A 선택 시): PC 절전·재부팅으로 장중 프로세스가 죽으면 해당일은 "오류일"이 아니라 "미가동일"로 기록하고 카운트에서 제외한다. 다만 미가동이 잦으면 연속성이 깨지므로 절전 해제 설정을 권장.

## Stage 2. G3 ― 페이퍼트레이딩 연속 10거래일

### 2.1 목표·판정 조건

실시간 시세로 가상 체결을 10거래일 연속 무오류 운용하고, 일일 리포트로 증적을 남긴다.

| G3 조건 | 검증 방법 |
|---|---|
| ① 페이퍼 연속 10거래일 무오류 | 운용일지 + 로그 (ERROR 0건 기준은 2.3 참조) |
| ② 백테스트 성과지표 산출 정상 | `scripts/run_backtest.py` 재실행 결과 보관 |
| ③ 일일 리포트 자동 생성 | 매 거래일 리포트 수신·기록 |

### 2.2 일일 운용 루틴 (부록 G 매핑)

| 시점 | 작업 | 명령/방법 |
|---|---|---|
| 장 시작 전 (08:30~) | 휴장일 확인 → PC/서버 가동 → 봇 기동 | `python -m src.orchestrator` → "▶ 기동 (mode=paper)" 알림 확인 |
| 장 시작 전 | kill switch 리셋 상태 확인 | 전일 발동했다면 원인 확인 후 재기동 (재기동 시 초기화됨) |
| 장 중 (수시) | 알림 채널 확인 | 주문·거부·헬스체크 실패 알림 여부 |
| 장 종료 후 (15:30+) | 안전 종료 (방식 A) | Ctrl+C → "■ 종료" 알림 확인 |
| 장 종료 후 | 당일 점검 | `scripts/healthcheck.sh`(서버) 또는 `logs/trader_YYYY-MM-DD.log`에서 `| ERROR` 검색 |
| 장 종료 후 | 운용일지 기입 | 부록 A 양식 (5분 이내로 간단히) |

### 2.3 "무오류"의 판정 기준

| 구분 | 처리 |
|---|---|
| ERROR 로그 0건 + 정상 기동·종료 | 정상일 ― 카운트 +1 |
| 시세 이상치·리스크 거부·신호 없음 | **정상 동작임** (시스템이 설계대로 보류·거부한 것) ― 카운트 +1 |
| 일시 네트워크 오류 후 자동 회복 (사이클 격리 동작) | 정상일로 카운트하되 일지에 기록, 3일 내 재발 시 원인 규명 |
| 프로세스 다운·미체결 방치·정합성 오류·헬스체크 연속 실패 | **오류일** ― 원인 규명·조치 후 **1일차부터 재시작** |
| 미가동일 (휴장·PC 사정) | 카운트 제외 (연속성 유지, 단 5거래일 이상 공백 시 재시작 권장) |

### 2.4 병행 작업 (선택)

- 주 1회 `scripts/backup.sh` 실행 (백업 습관화 ― G5 복구 리허설 대비)
- 페이퍼 성과와 백테스트 성과의 괴리 관찰 (슬리피지·체결 타이밍 차이 학습)
- SMA 교차는 검증용 예시 전략임을 재인지 ― G3 통과는 "파이프라인 신뢰" 확보이지 전략 수익성 검증이 아님

- [ ] 운용일지 10거래일 연속 "정상일" 기록
- [ ] 일일 리포트 10건 보관 (Telegram 수신 캡처 또는 DB)

## Stage 3. G4 ― 소액 실거래 1주 완전 사이클

### 3.1 목표·판정 조건

실주문 1주로 매수→체결→매도 전 과정을 검증하고, 수수료·세금 실제값으로 코드 상수를 확정한다.

| G4 조건 | 검증 방법 |
|---|---|
| ① 1주 매수→체결→매도 완전 사이클 | 실거래 1회 |
| ② 주문·체결·잔고 3자 일치 | 시스템 DB ↔ API 조회 ↔ 토스증권 앱 대사 |
| ③ 수수료·세금 실제값 일치 | 체결내역과 `estimate_cost` 계산값 대조 |

### 3.2 사전 조건 (전부 충족 시에만 착수)

- [ ] G3 확정 (익일 재확인 포함)
- [ ] 실시 일자: **시장 급변동일(지수 ±3%) 회피**, 장 초반 동시호가 직후·장 마감 직전 회피 (10:00~14:00 권장)
- [ ] 운용자가 장중 직접 관찰 가능한 날

### 3.3 단계별 절차

| Step | 작업 | 상세 절차 | 확인 사항 |
|---|---|---|---|

| 3-4 | 3자 대사 | ① DB `orders`/`executions` ② `scripts/check_account.py`는 **봇 정지 후** 실행 ③ 토스증권 앱 체결내역 | 수량·가격·상태 3곳 일치 |
| 3-5 | 비용 대사 | 앱 체결내역의 수수료와 `estimate_cost` 계산값 비교 | 수수료율(`/api/v1/commissions`) 일치 확인 |
| 3-6 | 매도 사이클 | 매도 신호 대기 또는 검증 목적의 수동 판단 매도 1주 (수동 시 봇 정지 후 실행·일지 기록) | 체결·잔고 반영 확인 |

| 3-8 | 판정 | `docs/gates.md` G4 기입 → 익일 재확인 | ― |

### 3.4 실패 시 조치 (즉시)

## Stage 4. G5 ― 소액 운용 20거래일 → 본격 운용

### 4.1 판정 조건

| G5 조건 | 검증 방법 |
|---|---|
| ① G4 후 소액 운용 20거래일 무사고 | 운용일지·일일 리포트 |
| ② 일일손실 한도 위반 0건 | risk_events 로그 (kill switch 발동 이력) |
| ③ 복구 절차 실동작 검증 | 4.3 리허설 2종 수행 |

### 4.2 운용 규칙

- 한도: G4의 최소 한도에서 소폭만 상향 (예: `max_order_amount` 30만~50만원 수준)
- 일일 루틴은 Stage 2와 동일 + **장 시작 전 reconcile 일치 확인** 추가
- "무사고" 기준: 정합성 불일치·미체결 방치·한도 위반·프로세스 다운으로 인한 포지션 방치 = 사고. 리스크 거부·시세 보류는 정상 동작
- 사고 발생 → 원인 규명·조치 → **20거래일 1일차부터 재시작**

### 4.3 복구 리허설 (20거래일 기간 중 1회씩, 비거래시간·포지션 없는 날 권장)

| 리허설 | 절차 | 합격 기준 |
|---|---|---|
| 강제 종료 복구 | 가동 중 프로세스 강제 종료(작업관리자/`kill -9`) → 재기동 | 재기동 시 reconcile 통과·정상 재개, 포지션 정합성 유지 |
| 백업 복원 | `scripts/backup.sh` 백업본을 별도 폴더에 복원 → DB 열람 | trade.db 조회 가능, 최근 이벤트 존재 |

### 4.4 본격 운용 전환 (G5 통과 후)

## 중단·에스컬레이션 기준 (전 Stage 공통)

다음 중 하나라도 발생하면 **즉시 정지 → 원인 규명 전 재개 금지**:

1. 정합성 불일치 (halted 발동) ― 자동 보정 금지, 수동 규명
2. 의도하지 않은 주문·중복 주문 발생
3. kill switch 발동 (당일 재개 금지, 익일 원인 검토 후 리셋)
4. Secret·토큰 노출 의심 ― 즉시 포털에서 재발급 + 전면 점검
5. API 스펙 변경 공지 ― `spec/openapi.json` 재다운로드·대조, dry-run 회귀 후 재개

재개 3조건(부록 E): ① 정합성 일치 ② 원인 규명·조치 완료 ③ dry-run 회귀 통과
